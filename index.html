Think like a programmer

Thinking like a programmer means we use:

  • Procedural Thinking: Breaking large goals into smaller steps that eventually get to the goal
  • Abstract Thinking: Think of a universal solution that could be applied to various similar situations, not just one specific situation.
  • Systems Thinking:
    • Systematic Debugging
    • Big Picture Thinking
    • Decision Making
  • Technological Empathy - I found this one most enchanting:

    Computers are “stupid”. We need to help the computer to understand what the problem is - speaking the language a computer would understand - so that the computer can do what we want it to do.

    Computer empathy is the ability to understand what a computer is, how it works, and what it's good and bad at doing.

    A computer is a tool, as are the programs we use and the programming languages used to write them. It's almost impossible to program anything substantial without a basic understanding of how these tools work.

  • Debugging:

    Debugging is a systematic process of relentlessly identifying the cause of a computer program that doesn't work.When a program doesn't work, it's because there is a mistake (also known as a "bug") somewhere in the computer code. Since these mistakes are an unavoidable part of programming, every good programmer has a system for fixing them and that system usually looks something like this:

    1. Collect evidence (what makes this program fail?)
    2. Generate theories (what may have caused this problem?)
    3. Test those theories (if my theory is correct, how could I find out?)
    4. Fix the problem

HTML Basics

How the "Web" is structured?

HTML is the basic document format that contains all the contents the author wants to share with the rest of the web.

Many HTML files are hyperlinked through URLs. By hyperlinking the URLs, we'll be able to navigate from one file to another, hence the "web".

HTTP is the internet protocol that allows my browser to access the files stored on the server.

HTML Basics

HTML files are written with tags and attributes.

The open tag and its close tag together with their enclosed content are called Element. Note that some tags are void tag which by itself is an element. Such as <br> or <img>. Some tages are inline tag, which stays in the line of the text. Some others are block tag, which create a rectangle box surrounding the text or image and block the content off the flowing line of the previous text. There's also another type of tag called Container Tags. Unlike 'paragraph element' or 'header 1 element' which specifies the importance or purpose of the text, the container tag is invisible, and doesn't serve any specific content but to just contain the elements.

DOM, Document Object Model, "defines the logical structure of documents and the way a document is accessed and manipulated."(W3 School).

HTML tags or elements are written in a nested structure, like a tree. We should carefully nesting the block elements inside the container elements, and inline elements inside the block elements. Since all the HTML elements are ensencially a rectangular box, when a front end developer receives a design mockup from the graphic designer, the first thing she does is to 'boxify' the mockup, which will serve as the basis of the HTML structure she would write later.

Why Use CSS

There're 3 ways to tell HTML what each of its elements should look like:

  1. Inside the element, define the style as element attribute:
    <h1 style="color: #555; font-size: 18px; font-style: italic;">Title</h1>
  2. Inside the <head> of the HTML file:
    <html>
	<head>
		<title>My Page</title>
		<style>
			h1 {
				color: #555;
				font-size: 18px;
				font-style: italic;
			}
		</style>
	</head>
	<body>
	...
  3. Write the CSS in a separate file, and link to that file from the HTML:
    <html>
	<head>
		<title>My Page</title>
		<link rel="stylesheet" type="text/css" href="style.css">
	</head>
	<body>
	...

The first method is not recommended. The last method is recommented.

When we write code, one basic principle is DRY, which stands for Don't Repeat Yourself. When you put all the style definition in each and every HTML element, understandably you'll ended up writing a lot of repeatitive code, like defining what the paragraph font should be, again and again, every time you have a new paragraph element. The second method is better than the first, because it allows you to define, for instance the <h1> once, then you'll be able to apply that same style to all the <h1> appeared in this HTML file.

However, the last method is the most recommended, especially when we're dealing with large websites where there're tons of HTML files that need to follow the same graphic style. By writing a separate CSS file, we can simply link to that same CSS from multiple HTML files. This way we avoided repeating ourselves.

The "Don't Repeat Yourself" principle is not just so that we can write less code; It's also important when we're debugging or dealing with changes. If the designer decided to change the color of all the hyperlinks in this website... I would be thanking myself for writing a separate CSS file.

Cascading Style Sheets

"Cascading" means that rules are applied not only to the elements they directly match, but also to all of those elements' child elements. However, if a child element has multiple, overlapping rules defined for it, the more specific rule takes effect.

This idea is called inheritance.

"Serious" Programming

The difference between
a toaster and a computer

A toaster has designated function - to toast. Even though you can add advanced features such as setting a timer, automatically adjusting the temperature based on thickness of the bread... etc., it’s function is still to toast.

A computer on the other hand, is an universal machine that can be programmed to perform different functions. A computer without any program installed on it is a dumb machine that can’t do anything. It doesn’t have any designated function. But programs do. Each and every program has a disgnate function to fulfill. Once a computer is programmed, it can do anything - any computational thing.

So what is a program? A program is a specifically and strictly defined sequence of steps. When we program, we're essentially designing the sequence of steps the computer needs to take to fulfill the function we want. The program communicates our ideas to the computer in the language a computer understands.

Natural Language vs. programming language

Natural language like English or Mandarin, is ambiguous and verbose. For instance the word 'biweekly' can mean either twice a week or every two weeks. Natual language relies a lot on the context to express its precise meaning, hence its verbosity.

Programming language follows a much more strict grammar, which doesn't leave any room for syntax error or subjective interpretation.

String Indexing in Python

Understand s[x] and s[x:y]:

  • The stirng s is considered as an array of characters.
  • In the first instance, x is the index position of that character. x belongs to range [0, s.length). If x goes beyond s.length-1, s[x] is undefined.
  • In the second instance, x and y can be any integer. If both x and y are within the string's range, and if x is smaller than y, s[x:y] outputs the characters between and include s[x] and s[y-1].
  • If x equals or is larger than y, or if x and y are out of the range of the string, then s[x:y] will output an empty string. It's still a valid expression.

The caveat here is, when you refer to a index range s[x:y] it will always be a valid expression, even if x or y is referring to a position that doesn’t exist in the string.

But when you refer to a position, s[x], x has to be an index position s does have. For instance,

s = ''
print s[0]    # Error

This would throw an error because s is empty, thus s[0] doesn’t exist.

Also pay attention to s[x] when x < 0. s[-1] is referring to the last letter in the string, when the string is not empty.

Find String

string.find(s, sub[, start[, end]])
string.find(s, i)      # i is any integer

Find the exact match s in string. It returns the lowest index position of the s. If a substring sub is provided, defined by the start and the end indice in the string, then this method will look for the target string s only in the substring.

.find method returns -1 if the target string is not found in the searched string. The searched string and target string has to match exactly, including the upper or lower cases...

print "text".find("text")     # prints 0
print "text".find("Text")     # prints -1
print "text".find("")         # prints 0
print "text".find(" ")        # prints -1

The example below shed light on how a simple “Find and Replace” function seen in all kinds of text editors might actually be programmed:

print "Example 3: using find to get rid of exclamation marks!!"
example = "Wow! Python is great! Don't you think?"
first = example.find('!')
second = example.find('!', first + 1)
new_string = example[:first] + example[first+1:second] + example[second+1:]
print new_string # oops, I should probably replace the !s with periods
new_string = example[:first] +'.'+ example[first+1:second] +'.'+ example[second+1:]
print new_string

Pay attention here how to use the 2nd argument in the function to find the occurences after the first one:

second = example.find('!', first + 1)

Input & Output

Input is what you feed to a procedure, or a function. Output is the result of the procedure.

When we define a function to produce some result, we need to write an expression starting with return. Otherwise the function won't give you any result, or output. 

def sum(a, b):
	a = a + b

print sum(5,3)         # It will print "None".

Also, function doesn't change the value of the input. Here's another example:

def sum(a,b):
	a = a + b
	return a

x, y = 5, 3
print (x, y)      # >>>8
print x           # >>>5

Procedure Composition

Procedure Composition means to use the output of one procudure as the input of another procedure. For example:

def sq(a):
	return a * a
print sq(sq(35))      # >>> (35^2)^2

Procedure Composition can lead to many interesting and complex results.
